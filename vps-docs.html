VPS Team Access Guide

Internal documentation for the Main Channel Brewing website project. This page covers how to SSH into the VPS, basic terminal navigation, where the site files live, how to edit files, how to create/delete folders and files, how to push changes to the GitHub repository, and what team privileges enable.

Reminder: This page is intended for the project team only. Treat server credentials and internal details as sensitive.

1. SSH Access from Windows

Windows 10/11 includes an SSH client built into PowerShell and Command Prompt.

Server IP
31.97.6.77
SSH Command
ssh <username>@31.97.6.77

Steps:

  1. Open PowerShell (recommended) or Command Prompt.
  2. Run the SSH command using your username.
  3. Enter your password when prompted (nothing will appear while typing — that’s normal).
# Example
ssh emmy@31.97.6.77

To disconnect when finished, type exit and press Enter.

2. Basic Linux Navigation Commands

Once logged in, you’ll work from the terminal. These commands are the basics for moving around and viewing files.

Where am I?

pwd

List files

ls
ls -la

Change directory

cd foldername
cd ..

Clear screen

clear

Tip: Use the Tab key to auto-complete file and folder names.

3. Website File Location

The website files are stored in the following directory on the VPS: 

/var/www/brewery-site

Navigate there after you log in:

cd /var/www/brewery-site
pwd
ls -la

If you see project files (HTML/CSS/JS/images), you’re in the right place.

4. Editing Files with nano

nano is a simple terminal text editor (great for quick edits).

Open a file to edit:

nano index.html

Save and exit:

  • Save: Press CTRL + O, then press Enter
  • Exit: Press CTRL + X
  • If asked to save changes, press Y (yes) or N (no)
Tip: If you’re editing a shared file, keep changes small and test them immediately after saving.

5. Creating and Managing Directories (Folders)

You may need to create folders for images, scripts, or other assets. Use mkdir to create folders, and use rm to remove files or directories when needed.

Create a folder:

mkdir images

Create nested folders in one command:

mkdir -p assets/images
mkdir -p assets/scripts

Verify files and folders:

ls -la

Delete a File

Use rm to delete a file. This is permanent (there is no recycle bin). 

rm filename.html

If you want to confirm before deletion, use interactive mode: 

rm -i filename.html

Delete a Directory (Folder)

If the folder is empty, use rmdir: 

rmdir foldername

If the folder contains files, use recursive delete. Be careful—this deletes everything inside permanently: 

rm -r foldername

If you want to be prompted before each removal (safer): 

rm -ri foldername
Safety tip: Before deleting, run ls -la and confirm you are in the correct directory. A good habit is to check pwd first.

The project directory is configured so new folders inherit the shared brewery group automatically.

6. Git + GitHub Workflow (Commit and Push Changes)

The website directory is connected to our GitHub repository. Use the workflow below to keep changes tracked and shareable. Prefer commits + pushes over “one-off” server edits when possible.

Before you start work (recommended every session):

cd /var/www/brewery-site
git status
git pull

Why pull first? It reduces merge conflicts and ensures you’re building on the latest team changes.

Create a new branch for your work (recommended):

git checkout -b feature/short-description

After you edit files, check what changed:

git status
git diff

Stage your changes:

git add .

If you only want to stage one file: git add path/to/file.html

Commit with a clear message:

git commit -m "Update merch page layout"

Push your branch to GitHub:

git push -u origin feature/short-description

Open a Pull Request (PR):

  • Go to GitHub in your browser.
  • Create a PR from your feature/... branch into main.
  • Request review from the team lead (Mark) or another reviewer.
Team rule of thumb: Use branches + PRs for anything beyond small text edits. It keeps history clean and reduces risk.

Common Git Commands (Quick Reference)

git branch                 # list branches
git checkout main          # switch to main
git pull                   # pull latest changes
git log --oneline --max-count=10

Common Issues and Fixes

1) “Permission denied” while pushing: You may need GitHub credentials/token, or your SSH key isn’t set up.

2) “Local changes would be overwritten by merge”: Commit or stash your work before pulling.

git status
git stash
git pull
git stash pop

3) Merge conflict: Git will mark the file. Edit it to resolve, then commit again.

git status
nano conflicted-file.html
git add conflicted-file.html
git commit -m "Resolve merge conflict"

7. Team Permissions and Privileges

The team has two privilege layers:

  • brewery group — allows the team to edit project files under /var/www/brewery-site without needing sudo for normal file changes.
  • sudo group — allows administrative tasks by prefixing commands with sudo.

Examples of sudo tasks:

sudo apt update
sudo apt install <package-name>
sudo systemctl restart nginx
Best practice: Use sudo only when required. Most project work should be done without sudo.

8. Future MariaDB Access (When Implemented)

When we add MariaDB, database access will be controlled separately from SSH/system access. Being in the sudo group allows installing and managing the MariaDB service at the OS level, but database changes are governed by MariaDB user accounts and grants.

What team members may be allowed to do (depending on granted DB roles):

  • View tables and data
  • Insert and update records
  • Create or modify tables (if granted)

We will create separate MariaDB accounts for team members and assign appropriate privileges once the database is installed.

9. Best Practices

  • Always confirm your current directory with pwd before editing or creating files.
  • Use ls -la to review what’s in a folder before making changes.
  • Pull before you push: git pull first to reduce conflicts.
  • Use branches + PRs for team changes to keep the main branch stable.
  • Avoid changing system configs unless you know exactly what the change does.
  • Use sudo only for installs, service restarts, and approved admin tasks.
  • Test your changes after editing (refresh the site, validate layout, check links).
  • Log out when finished: exit.
Suggestion: Once we finalize the deployment workflow, prefer making changes in a DEV environment and promoting to PROD rather than editing directly on PROD.
